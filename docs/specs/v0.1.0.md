# Glyim Programming Language
## Strategic Alignment & Vision Specification

**Document Status:** Draft
**Version:** 0.1.0
**Date:** 2025-04-10
**Owner:** Language Design Team

---

### 1. Vision & Strategic Alignment

**Vision Statement**
To create the first "Tier 1" systems programming language where high-level expressiveness, memory safety, and developer ergonomics coexist without compromise. Glyim unifies the rigorous type theory of Haskell, the memory safety of Rust, and the syntactic ergonomics of Python into a state-of-the-art compiler platform.

**Elevator Pitch (Moore)**
> For high-performance systems developers who are dissatisfied with the steep learning curve of Rust's borrow checker and the performance overhead of Haskell's runtime, Glyim is a new systems programming language. It provides an "invisible" borrow checker and tiered incremental compilation that feels like a scripting language, but compiles to machine code with Rust-level performance. Unlike existing languages, Glyim treats metaprogramming as a first-class citizen, allowing compile-time code execution within a unified compiler architecture.

### 2. Problem Statement & Business Context

**The Problem**
Modern systems programming is trapped in a "Trilemma":
1.  **C/C++:** High performance but manual memory management leads to security vulnerabilities and crashes.
2.  **Rust:** Memory safety guarantees, but the borrow checker is often a "battleground" between developer and compiler, and compile times are slow.
3.  **Python/Go:** Excellent ergonomics and tooling, but lack the low-level control and type safety required for high-performance or safety-critical systems.
4.  **Haskell/Idris:** Extremely powerful type systems (Type Classes, Dependent Types), but historically poor performance, tooling, and ergonomics.

**The Gap**
There is no language that offers **Linear Types** (Rust) for memory safety combined with **Hindley-Milner Type Inference** (Haskell) for expressiveness, wrapped in a **Significant Whitespace Syntax** (Python) for ergonomics. Developers are forced to choose between "safe but slow to build" (Rust) or "fast to build but unsafe" (C++).

**Why Now?**
Hardware performance gains are slowing; developer productivity is the new bottleneck. The industry is moving toward "language-led design" where the compiler is a platform for optimization (e.g., E-graphs), not just a translator.

### 3. Target Users & Stakeholders

**Primary User Classes**
*   **The High-Performance Backend Engineer:** Builds distributed systems, blockchains, or game engines. Needs Rust-like control but wants faster feedback loops.
*   **The Safety-Critical Developer:** Works in aerospace, medical devices, or fintech. Needs formal verification capabilities but cannot afford Ada/Rust developer friction.
*   **The Language Power User:** Loves metaprogramming (Lisp macros, Template Haskell). Wants compile-time generation capabilities that don't require a PhD to understand.

**Stakeholders**
*   **Open Source Community:** The primary driver of adoption. Requires excellent DX (LSP, `cargo`-like tooling).
*   **Systems Architects:** Looking for a "Tier 1" platform for building domain-specific languages (DSLs) on top of Glyim.

### 4. Needs & Value Proposition

**Top User Needs**
1.  **Invisible Safety:** "I want memory safety guarantees without fighting the compiler on every mutable reference."
2.  **Instant Feedback:** "I want language server (IDE) features that update in milliseconds, not seconds."
3.  **Compile-Time Power:** "I want to write code that writes code (Macros) using the same language, not a distinct C++ preprocessor."

**Value Proposition**
*   **Hybrid Type System:** Combines Algebraic Data Types (Haskell) with Affine Types (Rust), allowing for "Option" types and "Result" types to be ergonomic.
*   **Query-Based Compiler:** Architecture centered on `glyim-db` enables "Tiered Incremental Compilation," updating only the relevant parts of the dependency graph on file save.
*   **First-Class Metaprogramming:** `comptime` functions are standard Glyim functions executed via `glyim-mir-interp`, providing Type-Level Turing completeness without separate macro syntax.

### 5. Strategic Scope: Goals and Non-Goals

**Goals (In Scope for v1.0)**
1.  **G-001 (Performance):** Generated binaries must run within 5% performance delta of equivalent Rust/C++ code (via `glyim-codegen-llvm`).
2.  **G-002 (UX):** Language Server (LSP) must provide type-checking updates within 100ms for single-file changes (Tier 1 compilation).
3.  **G-003 (Safety):** Must enforce memory safety in safe code without a Garbage Collector (GC).
4.  **G-004 (Meta):** Support `comptime` function execution for compile-time logic and AST generation.

**Non-Goals (Explicitly Out of Scope)**
*   **NG-001:** We will **NOT** implement a Garbage Collector. Memory is managed via affine typing (RAII).
*   **NG-002:** We will **NOT** support C-ABI compatibility as a primary goal. FFI is possible, but Glyim is not a "drop-in C replacement."
*   **NG-003:** We will **NOT** prioritize "startup time" (time-to-`main`) over throughput for long-running server processes.
*   **NG-004:** We will **NOT** support dynamic code evaluation (e.g., `eval`) of source strings at runtime. Execution is static (AOT) or interpreted via MIR/Bytecode, but not dynamic string eval.

### 6. High-Level Architecture & Constraints

**Architectural Enablers**
To achieve the Goals, the project utilizes a modular, state-of-the-art architecture defined in `Cargo.toml`:
*   **Invisible Borrow Checker:** Achieved via **Dataflow Analysis** (`glyim-lifetime` + `glyim-borrowck`) using MIR (Mid-level IR), distinct from syntactic checks. This enables "Non-Lexical Lifetimes."
*   **Optimization via Equality Saturation:** The compiler utilizes `glyim-egraph` to optimize code mathematically, allowing high-level abstractions (Haskell-style) to compile down to efficient loops (Rust-style) automatically.
*   **Virtual File System (VFS):** `glyim-vfs` decouples the compiler from the disk, enabling robust remote compilation and IDE integration.

**Strategic Constraints**
*   **Rust Version:** The compiler infrastructure is locked to **Rust 1.94.1+** (`glyim-interner` uses specific performance features).
*   **LLVM Dependency:** v1.0 relies on `inkwell` for codegen (`glyim-codegen-llvm`).
*   **Module System:** Explicit support for a module system from the get-go, handling namespaces and visibility strictly (via `glyim-def-map`).

**Business Outcomes (OKRs)**

**Technical Success Metrics**

*   **Correctness:** Pass 99% of the Nix OS test suite (adapted for Glyim) via standard `glyim-test`.
*   **Coverage:** Code coverage of the standard library (`glyim-runtime`) > 90%.

### 8. Risks, Assumptions, and Open Questions

**Top Risks**
*   **Risk R-001 (Complexity):** The "Trinity" of Rust + Haskell + Python may result in a language too complex for the average user.
    *   *Mitigation:* Aggressive focus on "Invisible" defaults and compiler error messages (`glyim-diag`) that guide the user.
*   **Risk R-002 (Macro Hygiene):** First-class metaprogramming (`comptime`) can lead to "Macro hell" if hygiene (`glyim-hygiene`) is not perfect.
    *   *Mitigation:* Implement `SyntaxContext` tracking from day one (included in `glyim-span`).

**Key Assumptions**
*   **A-001:** Developers are willing to adopt a new toolchain if the LSP experience is significantly better than Rust's.
*   **A-002:** LLVM is a sufficient backend for optimization; we do not need a custom backend yet.

**Open Questions (Research Spikes)**
*   **OQ-001:** Can `glyim-mir-interp` execute compile-time code fast enough not to slow down the build?
*   **OQ-002:** Is the "Invisible Borrow Checker" (Polonius-style) feasible within the v1.0 timeline without freezing the language design?

### 10. Glossary
*   **Glyim:** The programming language being specified.
*   **Comptime:** Compile-time execution environment.
*   **MIR:** Mid-level Intermediate Representation (Typed, Monomorphized, Control-Flow Graph).
*   **E-Graph:** Equality Graph, used for optimization.
*   **Invisible Borrow Checker:** A borrow checker that operates via dataflow analysis rather than lexical lifetimes, minimizing developer friction.
*   **Tiered Compilation:** A strategy where the compiler provides fast, incomplete results (syntax/defs) before heavy, complete results (typecheck/codegen).
