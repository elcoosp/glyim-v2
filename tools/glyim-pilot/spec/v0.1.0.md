## 1. Vision Statement

Glyim Pilot eliminates the human bottleneck in multi-agent compiler development by automating the entire feedback loop — seeding prompts, extracting code from streaming responses, executing in isolated worktrees, and feeding structured results back — so that 20 parallel AI streams can run 24/7 across free-tier providers with zero manual intervention.

## 2. Elevator Pitch

For compiler engineers who manually copy-paste between chat UIs and terminals, Glyim Pilot is an autonomous dispatch system that orchestrates AI coding agents across free-tier providers, automatically extracts and executes their output, and enforces production-quality gates on every commit — unlike manual workflows, our system runs 20+ streams in parallel with compile-gated commits, automatic rate-limit failover, and mutation-tested verification, reducing a 2-week wave to an overnight run.

## 3. Problem Statement & Business Context

### The Problem

Developing the Glyim compiler requires implementing 20+ independent streams of work (lexer, parser, type system, MIR, codegen, etc.). Each stream follows a precise workflow:

1. Assemble a large prompt from 5+ reference documents and source files
2. Paste the prompt into an AI chat interface
3. Watch the streaming response for code blocks
4. Copy each code block, save as a script, execute in the correct worktree
5. If compilation fails, copy the error, paste it back, wait for a fix
6. If the provider rate-limits, wait and retry or switch providers
7. When done, run full verification (test, clippy, fmt, workspace check)
8. Commit, push, create PR

This loop runs 10–50 times per stream, across 20 streams, across multiple AI providers. A human operator becomes the bottleneck at step 4, managing context windows, tracking which worktree is which, and babysitting rate limits at 3 AM.

### Why Now

- Free-tier AI providers (DeepSeek, z.ai, Gemini, Grok, Mistral) now offer coding quality sufficient for compiler implementation
- These providers allow 2–3 concurrent conversations each, enabling 11+ parallel streams at zero cost
- Manual orchestration of 20 streams takes 2+ weeks of dedicated human attention
- Every manual step introduces risk: wrong worktree, missed code block, untested commit

### Business Context

Glyim is a from-scratch Rust compiler with 26 crates across 4 dependency waves. Waves must execute sequentially (Wave 2 depends on Wave 1), but streams within a wave run in parallel. The critical path is compiler correctness — every commit must compile, pass tests, and maintain locked public interfaces.

## 9. Operational Concept & High-Level Scenarios

### Concept of Operations

The developer starts the Pilot CLI server, runs a pre-flight check to verify provider login status, then dispatches a wave. The CLI assigns streams to provider slots, the Chrome extension opens tabs, seeds prompts, watches for code blocks, and sends them back to the CLI for execution. The CLI applies file operations, runs verification, commits on success, and feeds errors back on failure. The developer monitors progress via the CLI status dashboard and intervenes only when the system cannot recover automatically.

### Key Scenarios

**Scenario 1: Happy Path — Wave 1 Overnight Run**

The developer runs `glyim-pilot wave 1` at 10 PM. The system dispatches 8 streams across 4 providers. Each stream: seeds prompt → receives `glyim-ops` → applies files → runs commit gates → commits on success → feeds errors back on failure. By 8 AM, 7 of 8 streams have completed with PRs created. The 8th hit a persistent compile error and is paused for review.

**Scenario 2: Rate Limit Recovery**

DeepSeek rate-limits Stream S06 mid-response. The extension detects the error, notifies the CLI, which fails over to Grok (has available slots). A new tab opens, a failover prompt with progress summary is seeded, and S06 continues from where it left off.

**Scenario 3: Compile Error Loop**

Stream S03 produces code that doesn't compile. The `::COMMIT` gate catches the error. The trimmed compile error is fed back to the AI. The AI fixes the error with a `::REPLACE` directive and issues another `::COMMIT`. This loop continues up to 5 fix rounds before escalating to the developer.

**Scenario 4: Crash Recovery**

The developer's machine restarts overnight. On reboot, the developer runs `glyim-pilot serve`. The CLI reads the persistent state file, reconnects to existing Chrome tabs, and resumes all active sessions.

**Scenario 5: Self-Review Gate**

Stream S01 passes all commit gates and the AI issues `::DONE`. The done pipeline runs: dead code check passes, coverage passes at 89%, mutation testing passes at 82%. The self-review prompt is sent. The AI reviews its diff, finds an unnecessary `.clone()`, fixes it with `::REPLACE`, and issues `::APPROVED`. The stream is marked complete.

## 3. Functional Capabilities & Behavior

### 3.1 CAP-DISPATCH: Wave and Stream Dispatching

**REQ-FUNC-001**: The system shall accept a wave number and dispatch all streams in that wave to available provider slots, respecting concurrency limits per provider.

**REQ-FUNC-002**: When `glyim-pilot dispatch SXX` is invoked, the system shall create a git worktree at `../glyim-worktrees/stream-XX/` on branch `stream-XX/v0.1.0` from `main`.

**REQ-FUNC-003**: When all slots for a wave are occupied, the system shall queue remaining streams and dispatch them as slots become available.

**REQ-FUNC-004**: The system shall support three dispatch strategies: most-slots-first (default), round-robin, least-loaded, and manual assignment.

**REQ-FUNC-005**: The system shall prevent dispatching more concurrent streams to a provider than that provider's configured `max_concurrent` value.

**REQ-FUNC-006**: When a provider is in cooldown (rate-limited), the system shall not assign new streams to that provider until cooldown expires.

### 3.2 CAP-PROTOCOL: glyim-ops File Operations Protocol

**REQ-FUNC-007**: The system shall parse `glyim-ops` fenced code blocks from AI responses, supporting the directives: `::WRITE`, `::REPLACE`, `::DELETE`, `::COMMIT`, `::INCOMPLETE`, `::DONE`, `::APPROVED`.

**REQ-FUNC-008**: When `::WRITE <path>` is parsed, the system shall create or replace the file at `<path>` (relative to worktree root) with the content between the directive and `::END`.

**REQ-FUNC-009**: When `::REPLACE <path>` is parsed, the system shall locate the `---FIND---` text within the file, verify it appears exactly once, and replace it with the `---REPLACE---` text.

**REQ-FUNC-010**: When `::DELETE <path>` is parsed, the system shall delete the file at `<path>` from the worktree.

**REQ-FUNC-011**: When `::REPLACE` find text appears 0 times in the target file, the system shall reject the operation and send a structured error to the AI.

**REQ-FUNC-012**: When `::REPLACE` find text appears 2+ times in the target file, the system shall reject the operation and report the ambiguity count.

**REQ-FUNC-013**: The system shall validate that `::WRITE` and `::REPLACE` paths do not escape the worktree directory.

**REQ-FUNC-014**: When `::INCOMPLETE` is parsed, the system shall send a "continue" feedback message to the AI.

**REQ-FUNC-015**: When `::DONE` is parsed, the system shall initiate the done pipeline (Section 3.7).

**REQ-FUNC-016**: When `::APPROVED` is parsed after the self-review gate, the system shall finalize the stream (push, create PR).

### 3.3 CAP-COMMIT: Compile-Gated Commit System

**REQ-FUNC-017**: When `::COMMIT <message>` is parsed, the system shall execute the commit pipeline before committing.

**REQ-FUNC-018**: The commit pipeline shall execute sequentially: (1) `cargo fmt`, (2) `cargo check`, (3) `cargo clippy -D warnings`, (4) `cargo test`. All must pass.

**REQ-FUNC-019**: If any commit gate fails, the system shall NOT commit. Instead, it shall send a trimmed, targeted error message to the AI.

**REQ-FUNC-020**: If `cargo fmt` changes files, the system shall stage those changes automatically (they are included in the commit).

**REQ-FUNC-021**: When all commit gates pass, the system shall execute `git add -A` followed by `git commit -m "stream-XX: <message>"` in the worktree.

**REQ-FUNC-022**: The system shall commit ALL modified files in the worktree, including files from previous uncommitted directives.

**REQ-FUNC-023**: When the commit pipeline fails due to compile errors, the system shall increment a fix-round counter. If the counter exceeds the configured maximum (default: 5), the system shall make an emergency WIP commit and escalate to the developer.

**REQ-FUNC-024**: When a successful commit resets the fix-round counter to zero.

### 3.4 CAP-CONTEXT: Smart Context Injection

**REQ-FUNC-025**: The system shall assemble prompts using a tiered context model: Tier 1 (essential: master context, contracts, test instructions, file-ops skill), Tier 2 (owned files: full or smart-truncated), Tier 3 (dependency interfaces: pub signatures only), Tier 4 (usage sites: only if modifying locked interfaces).

**REQ-FUNC-026**: The system shall strip all orchestration instructions from the master context (worktree setup, git commands, cargo commands, PR assembly) before injection.

**REQ-FUNC-027**: The system shall replace the `plan-to-cat-scripts` output skill with the `file-ops` skill in the injected prompt.

**REQ-FUNC-028**: For files exceeding the configured line limit (default: 800), the system shall apply smart truncation: keep pub items, struct/enum definitions, and function signatures; replace implementation bodies with `...`.

**REQ-FUNC-029**: The system shall enforce per-provider token budgets (default: 15,000 tokens) and progressively trim lower-priority tiers to stay within budget.

**REQ-FUNC-030**: The system shall read `streams.json` to determine which files a stream owns, modifies, and creates, and include only those files at full fidelity.

**REQ-FUNC-031**: For dependency crates, the system shall extract only the `pub` signatures of requested items, not full implementations.

**REQ-FUNC-032**: The system shall include existing test file previews (mod.rs + first 30 lines) to inform the AI that it must append, not overwrite.

### 3.5 CAP-RETRY: Rate Limit Detection & Recovery

**REQ-FUNC-033**: The Chrome extension shall detect provider error states (rate limits, server busy, capacity) by checking: (1) error UI elements, (2) page text excluding assistant messages, (3) send button state.

**REQ-FUNC-034**: When a rate limit is detected, the extension shall notify the CLI with the error type and message.

**REQ-FUNC-035**: The CLI shall place the affected provider in cooldown for the configured duration (default: per-provider cooldown secs).

**REQ-FUNC-036**: If a rate limit occurs and an alternative provider has available slots, the CLI shall initiate failover: close the current tab, open a new tab on the alternative provider, and seed a failover prompt with progress summary.

**REQ-FUNC-037**: If no alternative provider is available, the CLI shall apply exponential backoff on the same provider (base: 60s, max: 300s).

**REQ-FUNC-038**: When a stream fails over to a new provider, the failover prompt shall include: (1) a progress summary (turns executed, commits made), (2) the original stream brief, (3) current file state context.

**REQ-FUNC-039**: The system shall not attempt more than 3 failover attempts per stream before escalating.

**REQ-FUNC-040**: When the AI's response text contains phrases like "rate limit" or "try again later" (as part of discussing the topic, not an actual error), the system shall NOT trigger rate-limit recovery. Only error UI elements and page-level text outside assistant messages shall trigger recovery.

### 3.6 CAP-FAILOVER: Provider Pool Management

**REQ-FUNC-041**: The system shall maintain a provider pool with configurable providers, each with: URL, concurrency limit, input/send selectors, error patterns, and cooldown settings.

**REQ-FUNC-042**: The system shall track active sessions per provider and enforce that the count never exceeds the provider's `max_concurrent`.

**REQ-FUNC-043**: When a stream completes or aborts, the system shall free the provider slot immediately.

**REQ-FUNC-044**: The system shall support adding new providers by editing `.glyim-pilot.toml` without code changes.

**REQ-FUNC-045**: Provider selector configurations shall be user-editable in `.glyim-pilot.toml` to handle DOM changes without rebuilding the extension.

### 3.7 CAP-GATES: Production Quality Gates

**REQ-FUNC-046**: The commit pipeline gates shall be: (1) cargo fmt (auto-fix), (2) cargo check, (3) cargo clippy -D warnings, (4) cargo test.

**REQ-FUNC-047**: The done pipeline gates shall be: (1) full commit pipeline, (2) dead code elimination check, (3) code coverage check (≥ 80%), (4) mutation testing (≥ 75% kill rate), (5) workspace check, (6) cargo audit, (7) self-review prompt.

**REQ-FUNC-048**: The done pipeline shall only execute after the AI emits `::DONE`.

**REQ-FUNC-049**: When a done gate fails, the system shall feed the specific error (coverage gaps, missed mutants, dead code warnings) back to the AI for a fix.

**REQ-FUNC-050**: After the AI fixes a done gate failure, the system shall re-run the full done pipeline (not just the failed gate).

**REQ-FUNC-051**: The self-review gate shall send the AI the full diff and a structured review checklist covering: edge cases, performance, error handling, API consistency.

**REQ-FUNC-052**: The system shall only mark a stream as COMPLETE after the AI emits `::APPROVED` following the self-review.

**REQ-FUNC-053**: Gate strictness shall be configurable: "relaxed" (fmt + check), "normal" (+ clippy + test), "strict" (+ banned patterns + architecture + contracts), "production" (+ coverage + mutation + self-review).

### 3.8 CAP-INTERACTION: Chrome Extension AI Chat Interaction

**REQ-FUNC-054**: The extension shall inject prompts into provider chat input fields using `document.execCommand('insertText')` with fallback to clipboard-based paste.

**REQ-FUNC-055**: The extension shall submit messages via the provider's send button click or Enter key dispatch.

**REQ-FUNC-056**: The extension shall monitor AI responses using `MutationObserver` with debounced extraction (500ms minimum interval).

**REQ-FUNC-057**: The extension shall extract code blocks from rendered `<pre><code>` elements, detecting complete blocks by the presence of closing triple backticks or the `::END` directive.

**REQ-FUNC-058**: The extension shall detect when the AI is still streaming (via provider-specific indicators) and not extract code blocks until streaming completes.

**REQ-FUNC-059**: Each tab shall run an independent content script instance; the background script shall route messages between the CLI and the correct tab.

**REQ-FUNC-060**: The extension shall not interact with tabs that are not registered as active pilot sessions.

### 3.9 CAP-RECOVERY: Crash Recovery

**REQ-FUNC-061**: The CLI shall persist session state to `.glyim-pilot-state.json` after every state change.

**REQ-FUNC-062**: On restart, the CLI shall read the state file and attempt to reattach to existing Chrome tabs by tab ID.

**REQ-FUNC-063**: If a tab is no longer available after crash recovery, the system shall queue the session for re-dispatch to an available provider.

**REQ-FUNC-064**: The system shall not re-execute scripts that were already committed; it shall resume from the last committed state.

### 3.10 CAP-STATUS: Monitoring & Dashboard

**REQ-FUNC-065**: The CLI shall display a real-time status table showing: stream ID, provider, status, turn count, and last activity timestamp.

**REQ-FUNC-066**: The CLI shall aggregate throughput metrics: turns per minute, total commits, total errors, total rate-limit recoveries.

**REQ-FUNC-067**: The CLI shall display a wave completion summary showing: stream ID, provider, turn count, fix count, and completion status.

## 1. Context & Scope

Glyim Pilot is a two-component system (CLI + Chrome extension) that automates AI-assisted compiler development. This document specifies the architecture, key decisions, and interface contracts.

**Key ASRs driving architecture:**
- ASR-01: End-to-end stream processing in ≤ 2s per turn (NFR-PERF-001)
- ASR-02: Commit pipeline ≤ 120s (NFR-PERF-003)
- ASR-03: Zero data loss on crash (NFR-REL-001, REL-004)
- ASR-04: Path traversal prevention (NFR-SEC-002)
- ASR-05: Local-only WebSocket (NFR-SEC-003)

## 2. Goals & Non-Goals

### Goals
- Support 5 providers with 11+ concurrent slots
- Process `glyim-ops` directives with sub-second parsing
- Execute commit pipeline automatically with 100% quality guarantee
- Recover from crashes with zero data loss
- Provide real-time status monitoring

### Non-Goals
- Multi-user deployment or shared session state
- Cloud execution or containerized sandboxes
- IDE integration
- Mobile device support
- Paid API access

### 3.5 Security Architecture

- **Path containment**: All file operations resolve paths relative to the worktree root and reject paths that escape it
- **Local-only WebSocket**: Server binds to `127.0.0.1:8420` only; no remote connections
- **No credential storage**: Extension never stores login credentials
- **Domain restriction**: Extension only activates on configured provider domains
- **Dangerous pattern detection**: Scripts containing `rm -rf`, `sudo`, `git push` require human confirmation

### 3.6 Deployment Architecture

- **CLI**: Standalone Rust binary, distributed as a single executable per platform
- **Extension**: Chrome Web Store package (or manual load via developer mode)
- **Configuration**: Single `.glyim-pilot.toml` in project root
- **State**: `.glyim-pilot-state.json` in project root (gitignored)
- **Worktrees**: `../glyim-worktrees/stream-XX/` as sibling directories

## 4. Architecture Decision Records

### ADR-0001: WebSocket for CLI-Extension Communication

**Status**: Accepted

**Context**: The CLI and Chrome extension need bidirectional real-time communication. Options: (A) Chrome Native Messaging, (B) HTTP + SSE, (C) WebSocket.

**Decision**: WebSocket on localhost.

**Alternatives Considered**:
- Native Messaging: Requires host manifest registration; platform-specific; harder to debug
- HTTP + SSE: Unidirectional from CLI; extension would need polling for CLI→extension messages

**Consequences**: 
- (+) Simple, cross-platform, debuggable with browser DevTools
- (+) Bidirectional, low latency
- (-) Requires CLI server to be running; no offline capability
- (-) Security limited to localhost binding

### ADR-0002: glyim-ops Protocol Over Bash Scripts

**Status**: Accepted

**Context**: AI agents produce code changes. Options: (A) Bash scripts with heredocs, (B) JSON operations, (C) Custom text protocol.

**Decision**: Custom text protocol (`glyim-ops`) with minimal directives.

**Alternatives Considered**:
- Bash scripts: 33%+ token waste on orchestration; fragile; security risks
- JSON: Verbose; hard for AI to generate correctly; poor readability

**Consequences**:
- (+) ~90% token reduction vs bash scripts
- (+) Surgical edits via REPLACE instead of full file rewrites
- (+) AI declares commit intent; pilot controls execution
- (-) Custom parser required
- (-) Not standard; AI must learn the protocol

### ADR-0003: Manifest V3 Chrome Extension

**Status**: Accepted

**Context**: Need to interact with provider chat UIs. Options: (A) Chrome extension, (B) Puppeteer/headless, (C) Electron app.

**Decision**: Chrome Manifest V3 extension.

**Alternatives Considered**:
- Puppeteer: No user session; can't use existing login cookies
- Electron: Heavyweight; overkill for tab management

**Consequences**:
- (+) Access to existing authenticated sessions
- (+) Lightweight; runs in background
- (-) Manifest V3 restrictions (no persistent background pages)
- (-) Chrome-only (MVP)
- (-) DOM-dependent; adapters break when providers change UIs

### ADR-0004: Compile-Gated Commits with AI Intent Declaration

**Status**: Accepted

**Context**: Commits must always compile. Options: (A) Auto-commit after every file operation, (B) AI declares commit intent, pilot verifies, (C) Manual commit only.

**Decision**: AI declares commit intent with `::COMMIT <message>`, pilot runs verification before committing.

**Consequences**:
- (+) 100% guarantee that committed code compiles
- (+) AI controls logical commit boundaries
- (+) Pilot batches multiple file operations into a single commit
- (-) AI may issue ::COMMIT too frequently or too rarely
- (-) Commit pipeline adds latency per turn

### ADR-0005: Git Worktree Isolation Per Stream

**Status**: Accepted

**Context**: Multiple streams modify the same repository. Options: (A) Separate clones, (B) Git worktrees, (C) Branches in same worktree.

**Decision**: Git worktrees, one per stream.

**Consequences**:
- (+) Shared .git objects (fast, disk-efficient)
- (+) Complete isolation between streams
- (+) Each worktree can have a different branch checked out
- (-) Requires git 2.5+
- (-) Worktree management adds complexity

## 6. Cross-Cutting Concerns

### 6.1 Observability

- CLI emits structured logs via `tracing` crate (trace/debug/info/warn/error levels)

- Status command provides real-time dashboard
- Session state file persists for crash recovery

### 6.2 Error Handling

- All errors use `anyhow::Result` with context
- Commit engine distinguishes: compile failed, test failed, clippy failed, pattern violation
- Rate limit handler classifies: rate_limit, server_busy, capacity, network_error
- Session state machine captures error state with recovery options

### 6.3 Testing Strategy

- Unit tests for parser, applier, context assembler, commit engine
- Integration tests for WebSocket protocol
- Mock provider adapters for end-to-end testing
- Property-based testing for parser edge cases

## 7. Alternatives Considered

### A: Puppeteer-based approach
**Rejected**: Cannot access existing authenticated sessions; requires managing cookies/tokens; heavyweight.

### B: API-based approach
**Rejected**: No free API access for target providers; paid APIs defeat the zero-cost goal.

**Rejected**: Overkill for tab management; high resource usage; harder to deploy than CLI + extension.

### D: Code-first with script generation
**Rejected**: 33%+ token waste; AI must generate correct bash; security risks from arbitrary script execution.

    And monitoring shall resume without data loss

  Scenario: Uncommitted files after crash
    Given a stream with uncommitted file changes
    When the CLI crashes
    Then the files shall be on disk (already written)
    And on restart, the CLI shall detect uncommitted changes
    And the next ::COMMIT shall include those changes
```

### 2.3 Mock Strategy

**MockCargo**: A test binary that simulates cargo commands:

- Accepts prompts
- Returns configurable `glyim-ops` responses
- Can simulate rate limits, server errors, and slow streaming
